# 04 · 核心技术实现要点

本文件聚焦几个最难、最关键的技术点，给后续开发（及 Codex）提供准确方向。

## 1. 高性能磁盘扫描

### 1.1 枚举文件的三档方案（按性能）
1. **`FileManager.enumerator(at:includingPropertiesForKeys:options:)`**：标准、易用。务必**预取**所需 `URLResourceKey`（如 `.totalFileAllocatedSizeKey`、`.isDirectoryKey`、`.contentModificationDateKey`、`.contentAccessDateKey`），避免逐个文件再发 stat 系统调用。
2. **`getattrlistbulk(2)`**：C API，一次系统调用批量取多文件属性，遍历海量小文件时显著更快。用于空间透镜全盘扫描的热路径。
3. **`fts(3)`**：C 的文件树遍历 API，也很快。可作为备选。

> 建议：常规模块用方案 1；空间透镜全盘扫描用方案 2 封装成 Swift。

### 1.2 体积口径（重要）
- **逻辑大小** vs **磁盘实际占用**：APFS 有压缩/克隆/稀疏文件，应优先用 `.totalFileAllocatedSizeKey`（实际分配）展示「能释放多少」，避免高估。
- 目录大小 = 递归累加子项实际分配大小。

### 1.3 并发与聚合
- `TaskGroup` 并行处理顶层目录；每个子任务把 (path, size, isDir, dates) 推入 `AsyncStream`。
- 一个 `actor TreeAggregator` 消费流、构建树、累加目录大小，天然无数据竞争。
- **有界并行**：`maxConcurrency ≈ min(核数, 6~8)`，避免 IO 抖动。
- **取消**：尊重 `Task.isCancelled`，用户取消时立即停。
- **进度**：按「已扫描字节 / 已知总量」或「已扫描项数」估算，UI 平滑显示。

### 1.4 内存控制
- 海量小文件不逐个长期持有：小于阈值的文件只累加进父目录汇总，仅保留**目录节点**和**大文件明细**。
- 节点用紧凑 `struct`（避免大量 class 引用开销），必要时用数组+索引代替指针树。

## 2. APFS 专属处理

- **可清除空间（Purgeable）**：APFS 把本地快照、缓存等算作「可清除」，`df` 里的可用空间可能含这部分。读取用：
  - `URLResourceValues` 的 `volumeAvailableCapacityForImportantUsageKey` / `volumeAvailableCapacityForOpportunisticUsageKey` / `volumeTotalCapacityKey`。
- **本地快照 / Time Machine 本地快照**：会吃掉大量空间，用户常困惑「明明删了东西空间没回来」。可检测并提示通过 `tmutil` 删除本地快照（需特权助手）。
- **APFS 克隆（Clone）**：复制文件可能是写时复制克隆，物理上共享存储。**重复文件检测必须识别克隆**，否则会把共享存储的「假重复」当成可释放空间（误导用户）。可通过比较 `inode` / 克隆关系判断。
- **快照感知**：删除大文件后空间可能因快照不立即释放，需向用户解释。

## 3. 清理规则引擎与定义库

### 3.1 定义库（Definitions）结构
把「什么是垃圾、是否安全」做成**数据驱动的可更新规则库**（JSON/SQLite），而非硬编码：

```jsonc
{
  "id": "user-caches-generic",
  "category": "system-junk",
  "title": "用户缓存",
  "description": "应用为加速重复操作而存的临时文件，删除安全，App 会自动重建。",
  "paths": ["~/Library/Caches/*"],
  "exclude": ["~/Library/Caches/com.apple.Safari/*"],   // 反例保护
  "safety": "safe",            // safe | caution | risky
  "defaultSelected": true,
  "sizeEstimator": "recursiveAllocatedSize",
  "requiresHelper": false,     // 是否需要 root
  "appRunningCheck": "com.example.app"  // 可选：运行中则警告
}
```

- **分级安全标签**：`safe`（默认勾选）/`caution`（默认不勾，提示）/`risky`（默认不勾，强警告，如语言包）。
- **可在线更新**：后台拉取最新定义库（带签名校验），让清理能力随新 App / 新系统进化，无需发版。
- **本地兜底**：内置一份基线定义库，离线可用。

### 3.2 语言包清理（高风险示例）
删除 App 内未用的 `.lproj` 本地化目录能省空间，但**可能破坏部分 App、且系统更新会重置**。必须：默认不勾、强警告、保留当前系统语言 + 英语兜底、可白名单。

### 3.3 开发者垃圾（差异化）
精确识别并安全清理：
- `~/Library/Developer/Xcode/DerivedData/*`
- `~/Library/Developer/Xcode/iOS DeviceSupport/*`（按版本）
- `~/Library/Developer/Xcode/Archives/*`（提醒：可能需要保留用于上架）
- `~/Library/Developer/CoreSimulator/`（不可用/旧运行时）
- Homebrew 缓存 `~/Library/Caches/Homebrew`、npm/yarn/pnpm、pip、CocoaPods 缓存
- Docker 悬空镜像（提示用户用 `docker system prune`，或调用）
- 扫描项目里的 `node_modules`（仅列出，删除交给用户确认）

## 4. 安全机制（SafetyEngine）实现要点

```swift
enum SafetyVerdict { case allow, deny(reason: String) }

protocol SafetyEngine {
    // 任何删除前必经此校验
    func verify(_ url: URL, intent: DeleteIntent) -> SafetyVerdict
}
```

强制规则：
1. **保护清单**：硬编码不可删根集合 `/System`, `/usr`(除 `/usr/local`), `/bin`, `/sbin`, `/private/var/db`(关键), 钥匙串, 用户 `~/Documents ~/Desktop ~/Pictures ~/Movies` 等非缓存根。
2. **路径规范化 + 越界防护**：`URL.standardized.resolvingSymlinksInPath()`，拒绝解析后落在保护区或卷根的路径；拒绝删除挂载点本身。
3. **删除即可恢复**：默认用 `FileManager.trashItem(at:resultingItemURL:)` 移入废纸篓（而非 `removeItem`），或移入应用隔离区，提供撤销。仅在用户明确「彻底删除」时永久删。
4. **特权操作白名单**：助手只接受「经主应用 SafetyEngine 校验过的路径列表」，且助手端**再次独立校验**（纵深防御，不信任客户端）。
5. **运行态检查**：清 App 缓存前用 `NSWorkspace.runningApplications` / bundleID 检测是否在运行。
6. **事务日志**：每次清理写 `~/Library/Application Support/Xico/logs` + 助手侧日志，记录删了什么、大小、时间、是否可恢复。

## 5. 卸载器（Uninstaller）

定位某 App 的全部关联文件：
- 主体：`/Applications/Xxx.app`（读其 `Info.plist` 取 `CFBundleIdentifier`）。
- 关联（按 bundleID + 名称启发式匹配）：
  - `~/Library/Application Support/<bundleID 或 名称>`
  - `~/Library/Caches/<bundleID>`
  - `~/Library/Preferences/<bundleID>.plist`
  - `~/Library/Containers/<bundleID>` & `~/Library/Group Containers/*<bundleID>*`
  - `~/Library/Saved Application State/<bundleID>.savedState`
  - `~/Library/Logs/<...>`、`~/Library/HTTPStorages/<bundleID>`
  - `~/Library/LaunchAgents/*<bundleID>*`、`/Library/LaunchAgents`、`/Library/LaunchDaemons`
  - 登录项、`/var/db/receipts/<bundleID>.*`（pkg 收据）
- **残留扫描**：反向找出 `Application Support`/`Caches` 中 bundleID 对应 App 已不存在的「孤儿」。
- 展示「将一并删除」清单 + 各自大小，全程可勾选预览。

## 6. 重复文件检测

三阶段，越往后越贵，逐级剪枝：
1. **按大小分组**：相同大小才可能重复（极快，纯元数据）。
2. **头部哈希**：对同大小组各取前 4–16KB 算快速哈希（**xxHash** 优于 SHA，速度快），剪掉绝大多数非重复。
3. **全量哈希或逐字节比对**：对仍同组者做完整内容哈希（xxHash/Blake3）或直接字节比较确认。
- **克隆识别**：APFS 克隆物理共享存储，**不计入「可释放」**，单独标注「克隆，删除不省空间」。
- **保留建议**：智能推荐保留（最早创建/最深目录/原位置），其余建议删除，但用户可改。
- **可选相似图片**：感知哈希（pHash）找视觉相似而非完全相同的图（Phase 2）。

## 7. 维护与优化（需特权助手）

- **维护脚本**：触发系统 `periodic daily/weekly/monthly`（清旧日志、轮转等）。
- **Spotlight 重建**：`mdutil -E`。
- **刷新 DNS**：`dscacheutil -flushcache` + `killall -HUP mDNSResponder`。
- **释放内存**：`purge`（需 root），或更克制地仅展示内存压力并建议。
- **释放可清除空间 / 删本地快照**：`tmutil deletelocalsnapshots`。
- **登录项管理**：读写用户/系统 LaunchAgents、`SMAppService` 登录项、`~/Library/LaunchAgents`。
- 这些全部封装为助手的**白名单命令**，绝不暴露任意命令执行。

## 8. 菜单栏监控（MenuBarExtra）

- SwiftUI `MenuBarExtra`（macOS 13+）。
- 指标采样：
  - CPU/负载：`host_processor_info` / `host_statistics`。
  - 内存压力：`vm_statistics64` / `host_statistics64`。
  - 磁盘：`URLResourceValues` 容量键。
  - 网络速率：读 `getifaddrs` 字节计数差分。
  - 电池：`IOKit` / `IOPSCopyPowerSourcesInfo`。
  - 温度/传感器：部分需特权或 SMC 读取（Apple Silicon 受限，谨慎）。
- 采样在后台低频进行（如 1–2s），空闲降频，杜绝自身成为耗电源。

## 9. 分发、签名、更新、合规

- **签名**：Developer ID Application 证书；主应用 + 助手同 Team ID；开启 **Hardened Runtime**；按需声明 entitlements（注意非沙盒）。
- **公证（Notarization）**：`notarytool` 提交苹果公证 + `stapler` 装订，保证 Gatekeeper 放行。
- **打包**：DMG（带拖入 Applications 的精美背景）或 PKG。
- **自动更新**：**Sparkle 2**，appcast 用 **EdDSA 签名**，HTTPS 分发。
- **许可 / 售卖**：FastSpring 或 Paddle（处理全球税务/VAT）；试用期 + 许可证校验；或订阅制。
- **隐私合规**：提供隐私政策；遥测匿名 + 默认可关；绝不上传用户文件内容或路径明细。

## 10. 测试策略

- **Domain 层单元测试**：注入内存型 mock 文件系统，覆盖 SafetyEngine（重点：保护清单、越界、符号链接逃逸的回归测试）、RulesEngine、扫描聚合。
- **安全回归测试**：构造「试图删保护路径/越界路径」的用例，断言一律被拒——这是发版红线。
- **集成测试**：在临时目录造假数据，跑真实扫描+清理（移废纸篓），校验结果与撤销。
- **XPC/助手测试**：协议契约测试 + 客户端身份校验测试。
- **性能基准**：大目录扫描耗时与内存基准，防回归。
- **UI 测试 / 快照测试**：关键页面深浅色快照。
