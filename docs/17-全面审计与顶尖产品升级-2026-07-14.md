## 4. 已发现并直接修复的关键问题

### 4.1 下载器与运行时组件

- 删除重复 Swift 源码，修复 P0 构建故障。
- 统一视频、音频、图片任务状态机；迟到回调不能覆盖暂停/取消；提前取消有门闩。
- 子进程取消先 SIGTERM，750ms 内未退出则 SIGKILL；输出泵和单行缓冲均有限额。
- 队列写入私有 0600 持久化文件；崩溃/重启后的进行中任务恢复为暂停，可由用户继续。
- 深链不再二次百分号解码；剪贴板媒体站点使用精确 host/子域关系，不做 substring 命中。
- 图片抓取补齐 HTTP 状态、超时、响应/实际大小、命名、取消和零文件失败语义。
- 新增 Ed25519 签名组件清单：序列号防回滚、签发/过期、最低 App 版本、组件 ID、版本、架构、HTTPS URL、SHA-256、精确大小、压缩格式和可执行名全部绑定。
- ffmpeg/aria2 必须来自签名清单；yt-dlp 在迁移期允许官方 URL + 官方 SHA2-256SUMS，配置清单后同样走签名目录。
- 安装前拒绝路径逃逸、绝对路径、控制字符、symlink、错误架构和非 Mach-O；通过全部校验后才清除 quarantine 并原子安装。

### 4.2 “不要拒绝危险输入”的最终口径

用户输入层不拒绝、不静默丢弃：任何非空输入都会生成可见任务。

1. 合法 HTTP、HTTPS、magnet 进入下载状态机。
2. `file:`、`javascript:`、带凭据 URL、畸形 magnet 等进入 `quarantined(reason:)`。
3. 隔离任务显示原输入、原因、盾牌状态和移除操作。
4. 隔离内容永不传给 yt-dlp、aria2、shell 或图片抓取器。

即：**接收危险输入并让用户看见，但不把它当成可执行指令。** 测试 `testDangerousDownloadInputIsAcceptedAndVisiblyQuarantined` 已证明这一契约。

### 4.3 SSH、SFTP、隧道与服务器

- 主机、用户名、端口和远端路径使用共享验证器；本地 Process 全程参数数组，不拼 shell 命令。
- 私钥、askpass、known_hosts 放入随机 0700 临时目录，文件 0600，任务结束即清理。
- 首次连接先扫描真实主机密钥，展示 endpoint、算法和 SHA256 指纹，由用户确认后固定。
- 后续连接强制 `StrictHostKeyChecking=yes`、私有 known_hosts、`UpdateHostKeys=no`；主机密钥变化直接阻断并提示重新验证。
- 跳板机先固定跳板机，再通过可信跳板扫描目标；两层身份分别绑定。
- SFTP 不再解析 `ls -la`：远端固定 Python 3 程序输出结构化 Base64 名称、类型、大小和权限。
- 换行、Tab、` -> ` 等特殊文件名不会破坏解析；无效 UTF-8/控制字符使用转义显示并只读，防止误操作。
- 目录只允许 `rmdir` 空目录；本地下载自动生成唯一文件名，不覆盖已有文件。
- SSH/SFTP 输出默认 16 MB、上限 64 MB；超限失败关闭，取消传播到子进程并在 750ms 内强停。
- 隧道修复启动/停止竞态、代次保护和强制终止回退。

### 4.4 智能扫描、系统垃圾与空间透镜

- 清理规则同时经过签名更新、摄入策略、路径红线、运行时 TOCTOU 复核和交叉重叠审计。
- 根目录、系统目录、用户内容根、云同步、Mail、凭据目录和符号链接逃逸均有拒绝路径。
- 高风险/个人数据类默认不选；失败不伪装为“0 项”；永久删除与废纸篓语义分开。
- 撤销覆盖同名冲突、双重撤销、废纸篓项丢失和部分恢复失败。
- 空间透镜按物理大小统计，硬链接只计一次，稀疏文件、符号链接、深层树和聚合桶均有回归测试。

### 4.5 性能与菜单监控

真实已安装旧构建曾测得约 12% CPU、96 MB footprint，触发本轮继续审计。当前代码完成：

- APFS 容量缓存 30 秒，消除每 2 秒重复卷查询；AppModel 复用同帧容量，不再二次读取。
- 只有当前状态栏样式真正需要时才枚举每核心、磁盘 I/O、swap 和 load average。
- IOHID 温度客户端/服务表进程内缓存；温度作为慢变量至少缓存 10 秒。
- 隐藏主窗口时，菜单栏快照使用独立 publisher；SwiftUI 页面不再因后台采样重排。
- 一帧二十余次 `@Published` 写入合并为一次 UI 发布；历史先落定、快照后发布。
- 相同签名返回同一 NSImage；状态栏不再把同一图像反复赋值给 NSButton。
- 健康分动画改为明确的一次性三阶段并回到静止值，不保留隐藏显示周期。

测量结果（2 秒档，CPU/内存/网络/磁盘/温度启用，主窗口关闭）：

| 构建/路径 | CPU | footprint | 说明 |
|---|---:|---:|---|
| 优化前已安装旧构建 | 约 12% | 约 96 MB | 只作问题线索；不是当前工作树 |
| 当前 Release 完整链路 | **0.269%** | **36.6 MB** | 7 帧；含采样、发布、历史/告警和 CG 字形栅格化 |

### 4.6 主题、配色、状态和无障碍

- 8 套主题集中展示，主题、明暗模式、Root tint/accent、按钮、开关和菜单监控代表色联动。
- 浅色语义文字色更新为 success `#087A52`、warning `#976000`、danger `#C5283D`、info `#2868C7`；对白对比度分别约 5.36、5.26、5.61、5.39:1。
- 深色语义色对暗背景的数值对比度均高于 5.1:1。
- 主按钮增加深色釉层以稳定白字可读性，禁用态、悬停、焦点环和破坏性按钮语义分离。
- 下载器、SFTP、服务器菜单和关键图标按钮补齐 accessibility label、action 语义和最小目标尺寸。
- 危险输入使用 warning/盾牌而不是“系统执行失败”式红色，准确表达“已接收但未执行”。

这些数值和代码可支持颜色对比判断，但本报告不以截图宣称完整 WCAG/VoiceOver 合规。

## 7. 发布与部署边界

`scripts/release_preflight.sh` 会把缺少这些配置视为发布阻塞；这属于部署凭据边界，不是运行时代码旁路。

## 9. 最终评价

Xico 的清理安全、下载信任链、服务器身份、恢复力、主题系统和菜单栏性能现已形成同一套可解释产品逻辑，而不是孤立功能堆叠。当前最强优势是：危险操作有清楚边界、异常状态不伪装成功、输入不被静默丢弃、用户始终知道系统做了什么。

剩余工作不再是已知 P0/P1 代码缺口，而是生产密钥/公证和真实外部环境认证。工作区原有未提交内容被保留，没有 reset 或覆盖用户其他改动。
