# Apple 公证(Notarization)问题 · 完整解决教程

> **重要更正（2026-07-10）：** 早前认为 status 7000 是"Apple 账号被风控/需联系 Apple 开通"——
> **这是误诊。** 真正原因是 `notarytool store-credentials` 填错了 `--team-id`：
> 用了 `AQ5TMWUPMH`，而它**不是 Team ID**，只是 Apple Development 调试证书 CN 括号里的
> 个人标识符。本账号真正且唯一的 Team ID 是 **`P22K8NF89K`**（Developer ID 证书的所有者）。
> 用错团队提交 → Apple 回 `statusCode 7000 "Team is not yet configured for notarization"`。
> **这是自己就能修的配置问题，无需联系 Apple。** 本文给出根因、修复、收尾三步。

---

## 一、根因（已核实）

| 来源 | CN 括号里显示 | **OU 字段 = 权威 Team ID** |
|---|---|---|
| Developer ID Application（签 App 用） | P22K8NF89K | **P22K8NF89K** |
| Apple Development（仅调试） | AQ5TMWUPMH | **P22K8NF89K** |

核对命令：

```bash
security find-certificate -c "Developer ID Application" -p | openssl x509 -noout -subject
# → subject= ... CN=Developer ID Application: Kai Yao (P22K8NF89K), OU=P22K8NF89K ...
```

- `security find-identity` 显示的是证书 **CN**，Apple Development 证书的 CN 括号里是**个人标识符**
  （`AQ5TMWUPMH`），不是团队号——这一点最容易误导人。
- 个人账号只有**一个**团队：`P22K8NF89K`。凡是给 `notarytool` 传的 `--team-id`，都必须是它。
- 代码签名本身一直是好的（硬化运行时 ✓、安全时间戳 ✓、无 get-task-allow ✓、`--deep --strict` 通过），
  **从来不是签名/打包问题。**

## 二、修复（一次性）

### 1. 用正确团队重建 notarytool 凭证

```bash
xcrun notarytool store-credentials "XicoNotary" \
  --apple-id "hi@yaokai.me" \
  --team-id "P22K8NF89K" \
  --password "<App专用密码>"
```

> ⚠︎ **`--password` 用的是「App 专用密码」**，格式 `xxxx-xxxx-xxxx-xxxx`（四组小写字母），
> 在 <https://account.apple.com> → 登录与安全 → App 专用密码 生成。
> **不要**用 App Store Connect 里那个 32 位十六进制的「App 专用共享密钥」——
> 那是**内购收据校验**用的，和公证无关，填进去只会认证失败。

成功会打印 `This process stores your credentials ... Credentials validated.`

### 2. 确认凭证可用

```bash
xcrun notarytool history --keychain-profile "XicoNotary"
```

### 3. 确认协议已签（本账号已确认「有效」）

在 App Store Connect → 商务 → 协议 里，「付费 App 协议」「免费 App 协议」应为**有效**。
另外去 <https://developer.apple.com/account> 看顶部有无「需接受更新的协议」横幅；有就点接受。
（若提交时报 `HTTP 403 — a required agreement is missing or has expired`，就是这一步没签。）

## 三、提交并收尾（一次跑完）

```bash
cd /Users/yaokai/Desktop/IT/MacApp/XicoApp

# 提交公证并等待（正常 Accepted 只要几分钟）
xcrun notarytool submit dist/Xico-Clean-0.2.7.dmg --keychain-profile "XicoNotary" --wait

# 若中途想看某次结果 / 失败日志：
#   xcrun notarytool info <id> --keychain-profile "XicoNotary"
#   xcrun notarytool log  <id> --keychain-profile "XicoNotary"

# Accepted 后：装订票据（让离线也能过 Gatekeeper）
xcrun stapler staple dist/Xico-Clean-0.2.7.dmg
xcrun stapler validate dist/Xico-Clean-0.2.7.dmg      # → The validate action worked!

# 终检：应显示 accepted · source=Notarized Developer ID
spctl -a -vv -t open --context context:primary-signature dist/Xico-Clean-0.2.7.dmg

# 上传新 DMG 到服务器
scp -i ~/Desktop/IT/web/Shangence.pem dist/Xico-Clean-0.2.7.dmg ec2-user@52.198.247.142:/tmp/
ssh -i ~/Desktop/IT/web/Shangence.pem ec2-user@52.198.247.142 \
  "sudo docker cp /tmp/Xico-Clean-0.2.7.dmg xicoai-app:/app/uploads/Xico-Clean.dmg && \
   sudo docker exec -u root xicoai-app chown nextjs:nodejs /app/uploads/Xico-Clean.dmg"
```

> 完整发布可直接跑 `scripts/notarize.sh`（构建→签名→公证→装订→打 DMG→公证 DMG→appcast），
> 该脚本的凭证注释已同步更正为 `P22K8NF89K`。

## 四、若真的仍卡在 In Progress / 7000（排除法）

先做完第二节（团队、密码、协议都对）后再看这里：

- **秒拒 7000**：`--team-id` 还是错的，或钥匙串里 `XicoNotary` 仍是旧的 AQ5TMWUPMH → 重跑第二节第 1 步覆盖。
- **~几十分钟后 403**：协议没签 → 第二节第 3 步。
- **>24–48h 仍 In Progress**：先看 Apple 系统状态页；若确认团队/协议都对仍长期卡住，
  才走 <https://developer.apple.com/contact/> 联系 Developer Programs Support，附 Team ID `P22K8NF89K`、
  7000 原文和卡住的 submission id。**注意：绝大多数情况根本走不到这一步。**

## 五、等待期间的临时分发（可选，不是必须）

公证不是**运行**的前提，只是免除 Gatekeeper 首次打开的警告。未公证的 Developer ID 签名 App 仍可分发：

- **右键打开**：拖入「应用程序」后，右键(或 Control 点)Xico.app → 打开 → 再点「打开」。
- **命令行去隔离**：`xattr -d com.apple.quarantine /Applications/Xico.app`
- **macOS 15+**：系统设置 → 隐私与安全性 → 底部「仍要打开」。

## 六、常见误区（不要做）

- ❌ **不要**因为看到 7000 就去联系 Apple 等开通——先检查 `--team-id` 是不是 P22K8NF89K；
- ❌ **不要**把「App 专用共享密钥」(32 位十六进制)当成「App 专用密码」填给 notarytool；
- ❌ **不要** `sudo spctl --master-disable`（全局关闭 Gatekeeper，危险且 Apple 禁止引导）；
- ❌ **不要**在配置修对前反复重交——只会堆积一堆卡住的 In Progress 任务。
