# Xico Precision Monitoring 全面升级设计

日期：2026-07-15

状态：首个实施切片完成

视觉方向：A「Precision Glass 精密玻璃」

## 1. 目标

把 Xico 的系统监控从“能显示数据”升级为可信、精致、可解释的专业监控产品。第一优先级是修复每款应用 CPU 与内存数据的准确度；随后让菜单栏面板、深度监控、设置、动效和 QA 使用同一套产品语言。

成功不是复刻 iStat Menus，而是同时做到：

1. 数值口径透明，可与 macOS `top`、活动监视器和受控负载交叉验证。
2. 多进程应用按应用聚合，同时允许展开查看子进程。
3. CPU 面板只以 CPU 为主叙事，内存面板只以内存为主叙事。
4. 两个面板的应用排行都同时显示 CPU 与内存，便于交叉判断。
5. 视觉比现有界面更克制、更紧凑、更有仪器感，颜色只承担数据语义。
6. 数据不完整、刚开始采样或权限不足时明确说明，不用貌似精确的错误值填充。

## 2. 范围与交付顺序

这是一个产品级项目，拆成四个有明确边界的工作流，按顺序交付：

### 2.1 Trust Core：可信采样基座

- 全量 PID 枚举、进程身份、CPU 差分和物理内存采样。
- 应用与子进程归属解析、应用级聚合、特权进程覆盖。
- 覆盖率、数据新鲜度、采样状态和错误模型。
- 准确度基准、压力测试和性能预算。

### 2.2 Menu Surfaces：菜单栏全套面板

- CPU、内存、网络、磁盘、GPU、温度、电池。
- 可配置菜单栏状态项、排行数量、密度、刷新频率和单位。
- CPU 与内存先完成，因为它们共享新的应用级采样模型。

### 2.3 Deep Monitor：深度监控中心

- 总览、单指标详情、应用检查器、跨指标趋势和异常解释。
- 硬件、传感器、网络接口、磁盘卷和 GPU 详情。
- 从任意应用行进入同一个应用检查器。

### 2.4 Product Polish：产品级精修

- Precision Glass 设计系统、主题、动效、无障碍和本地化。
- 暗色/亮色、不同尺寸、低动态效果和高对比度适配。
- 截图回归、能耗、采样开销、稳定性和发布前验收。

本文定义四个工作流的统一边界，并完整定义首个可实施切片：Trust Core + CPU/内存菜单面板 + 应用检查器基础入口。其余面板复用相同组件与验收标准，分批实施，避免一次性改动所有监控代码。

## 3. 已确认的产品规则

### 3.1 CPU 面板

CPU 面板的主指标只展示 CPU 相关信息：

- 总使用率、用户、系统、空闲。
- 性能核与能效核负载、频率和每核状态。
- 系统负载、温度/限频状态和 CPU 历史。
- 应用排行默认按 CPU 降序。

应用行同时显示：应用图标、应用名、进程数、CPU、物理内存。CPU 是高强调列，内存是辅助列。

### 3.2 内存面板

内存面板的主指标只展示内存相关信息：

- 内存压力状态与 Xico 压力指数。
- 物理占用、应用内存、联动/有线内存、压缩、缓存、可用。
- 换入、换出、交换区使用量与变化速率。
- 内存压力、压缩和交换历史。
- 应用排行默认按物理内存降序。

应用行同时显示：应用图标、应用名、进程数、物理内存、占总内存百分比、CPU。内存是高强调列，CPU 是辅助列。

### 3.3 交叉指标的边界

CPU 和内存不会在两个面板顶部混成一个综合分数。交叉指标只出现在：

- 应用排行，用于快速判断高 CPU 应用是否也占用大量内存。
- 应用检查器，用于查看同一应用的 CPU、内存、磁盘、网络和能耗趋势。

## 4. 当前问题与根因

### 4.1 PID 枚举错误

当前 `ProcessSampler.allPIDs()` 把 `proc_listallpids` 的返回值当成字节数，又除以 `MemoryLayout<Int32>.size`。该 API 返回的是 PID 数量，因此实际只遍历约四分之一的进程。结果会漏掉大量应用和高消耗进程，直接造成排行不可信。

### 4.2 CPU 口径不同

当前 Xico 的单进程 CPU 是按逻辑核心累加的原始口径，8 核机器理论上范围为 0–800%。用户对比的 iStat 配置使用 0–100% 归一化口径，因此同一负载可相差最多 8 倍。

### 4.3 没有应用级聚合

当前列表按 PID 展示。Chrome、Codex、Xcode 等应用会拆成主进程、Renderer、Helper、XPC 等多行，既难读，也无法反映整个应用的真实占用。

### 4.4 首帧与陈旧数据

进程排行只在详情消费者可见时采样。重新打开面板时，旧列表可能继续显示；首个 CPU 差分还可能跨越很长时间窗口，形成失真值。

### 4.5 权限覆盖不透明

本机有一部分系统进程会对普通进程调用返回 `EPERM`。Xico 已有 root helper，但监控采样尚未使用它。当前界面也没有说明数据覆盖率，用户会误以为列表是全量结果。

## 5. 数据架构

### 5.1 组件边界

```text
ProcessSnapshotProvider
        ↓
ProcessIdentityResolver
        ↓
ApplicationOwnershipResolver
        ↓
ApplicationUsageAggregator
        ↓
UsageRankingStore
        ↓
CPU / Memory Panels + App Inspector
```

#### ProcessSnapshotProvider

一次采样返回系统中可见 PID 的原始快照。它不负责应用归属或 UI 排序。

提供两个实现：

- `LocalProcessSnapshotProvider`：普通权限，始终可用。
- `HelperProcessSnapshotProvider`：通过现有签名校验 XPC helper 获取受保护进程的最小指标集。

协调器优先使用 helper；helper 缺失、版本不兼容或超时时退回本地提供器，并携带覆盖状态。

#### ProcessIdentityResolver

进程键必须是 `pid + startTime`，不能只用 PID。这样可以避免 PID 被系统复用后继承旧 CPU 基线。

解析字段：

- PID、父 PID、进程开始时间。
- 可执行文件路径、短名称。
- bundle identifier、bundle 根路径和图标候选。
- CPU 累计时间、物理内存 footprint、峰值 footprint。

#### ApplicationOwnershipResolver

把进程映射到“用户理解的一款软件”：

1. 优先使用进程自身 bundle identifier 与 bundle 根路径。
2. Helper、Renderer、XPC 等无独立主应用的进程沿父进程链寻找最近的 `.app` 根。
3. 已知 browser/electron/webkit 子进程按 bundle 根与审计 token 归属主应用。
4. 独立系统 daemon 保持独立，不强行归到无关 GUI 应用。
5. 无法确定归属时使用可执行文件身份，标记为“独立进程”。

解析结果缓存，但缓存键包含开始时间；进程退出后延迟清理，防止频繁 churn 造成抖动。

#### ApplicationUsageAggregator

对同一应用组的成员做同一采样时刻的聚合：

- `cpuRawPercent = Σ member.cpuRawPercent`
- `cpuNormalizedPercent = min(100, cpuRawPercent / logicalCPUCount)`
- `physicalFootprint = Σ member.ri_phys_footprint`
- `peakFootprint = Σ member.ri_lifetime_max_phys_footprint`
- `memberCount = 当前有效成员数`

默认显示归一化 CPU 0–100%。设置允许切换“总核心 0–N×100%”，但排序始终使用未截断原始值，避免两个满载应用被同时显示为 100% 后失去顺序。

#### UsageRankingStore

保存最新快照、短期趋势和状态，不负责系统调用。它提供：

- CPU 排行、内存排行。
- 最近 60 秒应用趋势。
- 快照年龄、覆盖率和采样状态。
- 选择某应用后需要的成员明细。

## 6. 采样语义

### 6.1 PID 枚举

- 把 `proc_listallpids` 返回值解释为 PID 数量。
- 第一次查询容量后分配带余量的缓冲区，再调用获取数据。
- 若结果达到容量上限，扩大缓冲并重试，处理采样期间的进程 churn。
- 过滤 PID 0、负数、重复项和采样中已退出的进程。

### 6.2 CPU

- 使用进程累计 user + system CPU 时间做相邻快照差分。
- 使用单调时钟计算 elapsed，避免系统时间调整影响结果。
- 首次看到某个 `pid + startTime` 时只有基线，CPU 状态为“采样中”，不显示 0.0% 冒充实测值。
- elapsed 超出允许窗口、累计值回退或身份改变时丢弃差分并重建基线。
- 菜单打开后先立即显示内存，CPU 在第二个有效样本到达时出现。

### 6.3 应用内存

应用排行采用 `proc_pid_rusage` 的 `ri_phys_footprint`。这是进程对物理内存压力的有效 footprint，和 macOS `top` 的 `MEM` 口径最接近。

明确不使用：

- 虚拟地址空间大小作为“内存占用”。
- 单纯 RSS 替代 footprint。
- 把系统级“应用内存”按比例摊给进程。

应用组 footprint 等于成员 footprint 的同刻求和。由于系统共享页、受保护进程和内核分类，排行各行之和不要求等于系统级“应用内存”；界面帮助文案必须说明这一点。

### 6.4 系统内存

系统级内存遵循 macOS 的分类概念：

- 应用内存：应用与进程使用的内存分类。
- 联动/有线内存：不可压缩或换出的系统内存。
- 已压缩：压缩器保存的物理内存。
- 缓存文件：可在需要时回收的文件缓存。
- 可用：当前可立即使用或可快速回收的内存。
- 物理占用：总内存减去可用，不与“应用内存”混为一谈。

“内存压力”优先显示正常/注意/严重三态。若显示数值，名称固定为“Xico 压力指数”，范围 0–100，并通过 tooltip 说明它是基于压力级别、可用量、压缩和 swap 活动计算的解释性指数，不伪装成 macOS 直接提供的百分比。

### 6.5 覆盖与新鲜度

每次快照携带：

- 枚举 PID 数。
- 成功采样数、权限拒绝数、采样期间退出数。
- 数据源：helper / local fallback。
- 采样时间和快照年龄。

正常状态不打扰用户。覆盖明显不足、helper 不可用或数据超过两个刷新周期时，在排行标题旁显示低干扰状态标识，点击可查看原因。

## 7. Helper 扩展

在现有 helper protocol 中加入只读批量采样接口。请求只传 PID 与所需字段掩码；响应只返回进程身份和资源数值。

安全约束：

- 复用现有代码签名、Team ID 和客户端校验。
- 不返回环境变量、命令行参数、打开文件或用户内容。
- 限制 PID 数量、请求频率和最大响应体。
- 设定短超时；失败不阻塞 UI，自动使用本地结果。
- helper 版本不匹配时显示可恢复状态，不让菜单崩溃或永久转圈。

## 8. Precision Glass 视觉系统

### 8.1 视觉原则

- 石墨黑与自适应玻璃承载结构，避免大面积彩色背景。
- 颜色固定语义，不随主题任意交换：CPU 蓝、内存紫、网络青绿、系统粉、磁盘琥珀、温度珊瑚红、健康状态绿色。
- 关键数字使用等宽数字特性，排行数值纵向对齐。
- 玻璃、阴影和辉光只用于层级，不遮蔽文字或图表。
- 支持 Reduce Transparency、Increase Contrast 和 Reduce Motion。

### 8.2 菜单尺寸与密度

- 默认逻辑宽度 336 pt，内容在常见 13 英寸屏幕上无需横向滚动。
- 密度设置：紧凑、均衡、详细；默认“均衡”。
- 应用排行默认 6 行，可选 4、6、10、20。
- 面板过高时内部内容分段折叠，而不是缩小字体。

### 8.3 CPU 菜单结构

1. 标题、芯片、实时/陈旧状态。
2. CPU 总环：用户、系统、空闲。
3. P/E 核频率与负载；详细模式显示每核小型热度环。
4. CPU 历史，支持实时、15 分钟、1 小时。
5. 应用排行：`应用 | CPU | 内存`。
6. 打开监控器、刷新/暂停与电源入口。

### 8.4 内存菜单结构

1. 标题、物理总量、实时/陈旧状态。
2. 内存压力状态 + Xico 压力指数；物理占用独立显示。
3. 应用、联动、压缩、缓存、可用的分段条与精确数值。
4. 换入、换出、swap 使用量和趋势提示。
5. 内存历史，切换压力、压缩、swap。
6. 应用排行：`应用 | 内存 | CPU`。
7. 打开监控器与安全清理入口；清理动作绝不暗示可以“释放所有占用”。

### 8.5 应用行与应用检查器

应用行点击后打开检查器，而不是立刻执行危险操作。检查器包含：

- 当前 CPU、物理 footprint、峰值 footprint。
- 60 秒与 15 分钟趋势。
- 子进程树与每个成员的 CPU/内存。
- 可用时显示磁盘 I/O、网络吞吐、唤醒和能耗信息。
- 数据来源、采样时间和覆盖说明。

退出或强制退出放在二级操作中，必须确认，并明确说明未保存数据风险。

## 9. 状态与错误处理

### 9.1 明确状态

- `warmingUp`：内存可显示，CPU 标注“采样中”。
- `live`：两个有效连续样本，显示实时数据。
- `partial`：部分 PID 无权限或采样失败，显示覆盖说明。
- `stale`：超过两个刷新周期未更新，保留最后值但加陈旧标识。
- `unavailable`：没有可用快照，显示原因和重试，不显示伪造的 0。

### 9.2 进程 churn

进程在枚举和采样之间退出是正常现象，记录为 churn，不当作用户可见错误。只有连续多个周期出现异常高失败率才升级为 partial。

### 9.3 排行稳定性

- 使用轻量滞后和稳定排序，避免两个数值接近的应用每秒互换位置。
- 排序主键变化超过可见阈值时立即更新。
- 退出应用淡出一帧后移除；新应用淡入，不复用旧趋势。

## 10. 刷新与性能预算

- 面板可见：默认 1 Hz；用户可选 1、2、5 秒。
- 深度监控可见：1 Hz；图表绘制与采样解耦。
- 仅菜单栏状态项可见：使用轻量系统总量采样，不强制全进程排行。
- 所有相关界面隐藏：停止高成本应用级采样，保留低频系统历史。
- 多个消费者共享同一快照，不重复扫描 PID。

基准目标（M1、约 1,200 PID）：

- 本地全进程采样 P95 小于 15 ms。
- 1 Hz 应用级采样给 Xico 增加的 CPU 小于 1.5 个百分点。
- 排行与 60 秒趋势缓存新增内存小于 12 MB。
- 打开菜单到内存首帧小于 150 ms；CPU 有效值在下一采样周期内出现。

## 11. 设置

新增“监控精度与显示”设置组：

- CPU 口径：0–100%（默认）/ 0–N×100%。
- 应用合并：默认开启；关闭后按独立进程显示。
- 子进程：默认折叠，可在应用检查器展开。
- 排行数量：4 / 6 / 10 / 20。
- 刷新频率：1 / 2 / 5 秒。
- 单位：十进制 GB / 二进制 GiB；默认跟随现有产品约定。
- 密度：紧凑 / 均衡 / 详细。
- 高级说明：显示每个指标的定义与数据来源。

设置变更立即生效，但 CPU 口径切换不重置原始采样基线。

## 12. 测试与验收

### 12.1 单元测试

- `proc_listallpids` 返回“数量”而非“字节”的回归测试。
- 缓冲区满时扩容重试、去重、过滤无效 PID。
- CPU 差分、单调时间、首次采样、超长间隔和计数器回退。
- PID 复用：相同 PID、不同开始时间不得继承基线。
- 应用归属：普通 App、Electron、Chrome Helper、XPC、daemon、孤儿进程。
- 应用 CPU/内存求和与两种 CPU 显示口径。
- 内存排序、CPU 排序、稳定排序和退出处理。
- live/partial/stale/unavailable 状态机。

### 12.2 集成测试

- 对稳定测试进程，Xico footprint 与 `/usr/bin/top` 的 `MEM` 在统一单位后差异不超过 5%。
- 受控单核负载下验证 raw 与 normalized CPU 的转换。
- 启动多子进程 fixture，应用聚合值等于所有成员之和，误差仅来自显示舍入。
- helper 存在、缺失、超时、版本不匹配和 `EPERM` 场景。
- 高频创建/退出进程，确保无崩溃、无越界、无旧 PID 趋势污染。

### 12.3 UI 与可访问性测试

- CPU 面板顶部不出现内存总览；内存面板顶部不出现 CPU 总览。
- 两个应用排行同时显示 CPU 和内存，且主列强调符合面板语义。
- 暗色、亮色、Reduce Transparency、Increase Contrast、Reduce Motion 截图回归。
- 336 pt 默认宽度及所有本地化语言不截断关键数值。
- VoiceOver 顺序、指标单位、趋势描述和状态说明完整。

### 12.4 完成标准

首个实施切片只有同时满足以下条件才算完成：

1. PID 枚举不再只采样四分之一。
2. CPU 默认口径与所选 iStat 0–100% 模式可直接理解地对齐。
3. 多进程应用默认合并，且可展开成员。
4. CPU/内存两个排行均同时显示两项数据。
5. 内存使用 `ri_phys_footprint`，系统内存分类和压力指数有清晰定义。
6. helper 缺失或部分失败时 UI 不撒谎、不永久转圈。
7. 准确度、性能和 UI 回归测试全部通过。

## 13. 后续面板如何超越 iStat

Trust Core 完成后，其余面板遵循相同模式：单一主叙事 + 应用交叉指标 + 可解释状态 + 深度检查器。

- 网络：实时上下行、接口、Wi‑Fi、延迟/丢包、按应用带宽与 VPN 状态。
- 磁盘：读写、卷空间、SMART、温度、TRIM、按应用 I/O。
- GPU：利用率、频率、温度、显存/共享内存、图形应用排行。
- 温度与风扇：友好传感器命名、来源分组、阈值、趋势和降频解释。
- 电池与能耗：健康、循环、功率、充放电、应用能耗与唤醒。
- 总览：只呈现跨指标异常与趋势，不重复堆叠各详情页全部数据。

Xico 的差异化能力是统一“应用检查器”：用户从任意排行进入同一应用，查看 CPU、内存、网络、磁盘和能耗如何同时变化，而不是在多个孤立菜单里寻找同一个应用。

## 14. 参考口径

- Apple Activity Monitor：内存压力、应用内存、联动内存、压缩和缓存定义。
- macOS `/usr/bin/top`：进程 `MEM` 为物理内存 footprint。
- Bjango iStat Menus：默认合并应用子进程、支持 CPU 0–100% 与 0–N×100% 显示口径。
