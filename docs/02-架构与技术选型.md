# 02 · 技术选型与系统架构

## 1. 技术选型分析（你最关心的「用什么技术好」）

### 1.1 结论先行

| 维度 | 选择 |
|------|------|
| 语言 | **Swift 6**（开启严格并发检查 Strict Concurrency） |
| UI 框架 | **SwiftUI 为主**，少量 **AppKit** 通过 `NSViewRepresentable` 桥接 |
| 最低系统 | **macOS 13 Ventura**（为使用 `SMAppService`），主力 14/15+ |
| 架构 | Apple Silicon + Intel **Universal Binary** |
| 分发方式 | **Developer ID 直接分发 + 公证（Notarization）**，非沙盒、非 App Store |
| 特权操作 | **`SMAppService` 注册的特权助手守护进程 + XPC** |
| 自动更新 | **Sparkle 2**（EdDSA 签名 appcast） |
| 持久化 | **SwiftData 或 Core Data**（设置/历史）+ **SQLite/JSON**（清理规则库） |
| 状态管理 | **MVVM + `@Observable`**（或类 TCA 单向数据流） |

### 1.2 为什么是原生 Swift，而不是 Electron / Tauri / Flutter

这类「磁盘清理器」是**对系统集成、性能、信任感要求极高**的应用，跨平台 Web 技术是错误选择：

| 对比项 | 原生 Swift/SwiftUI ✅ | Electron ❌ | Tauri ⚠️ | Flutter ⚠️ |
|--------|----------------------|-------------|----------|------------|
| 深度系统访问（FDA、特权助手、XPC、APFS） | 一等公民 | 需大量原生插件 | 需大量 Rust/原生桥接 | 需大量原生通道 |
| 扫描百万文件的性能 | 直接调底层 syscall，最快 | JS 层瓶颈明显 | Rust 可，但仍需桥接 | Dart 尚可但 IO 桥接复杂 |
| 原生质感 / 动效 / 毛玻璃 / 菜单栏 | 完美（Material、Metal） | 模拟，违和 | 用系统 WebView，质感一般 | 自绘，非原生味 |
| 安装包体积 | 小（~20–40MB） | 大（150MB+） | 中 | 中 |
| 电池 / 内存友好 | 优 | 差（常驻工具致命） | 中 | 中 |
| 用户对「清理器」的信任 | 原生=可信 | 体验廉价感 | 一般 | 一般 |

> CleanMyMac X、DaisyDisk、OnyX 等同类标杆**全部是原生**。一个常驻、要删你文件、要 root 权限的工具，用 Electron 是反例。**坚决原生。**

唯一例外：如果未来要做配套**官网 / 落地页 / 账户中心**，那是 Web 的活，与 App 无关。

### 1.3 关键第三方库（克制使用）

| 库 | 用途 | 备注 |
|----|------|------|
| **Sparkle 2** | App 自动更新（非 App Store 必备） | 成熟标准方案 |
| **Swift Collections** | 高效数据结构（OrderedSet 等） | Apple 官方 |
| **Swift Async Algorithms** | 异步流处理扫描结果 | Apple 官方 |
| 自绘 / Charts | 空间透镜可视化 | 优先自绘 Canvas/Metal，复杂图表可用 Swift Charts |
| **FastSpring / Paddle SDK** | 售卖与许可证（含税务合规） | 商业化阶段接入 |

> 原则：**能用系统框架就不引第三方**。清理器对依赖供应链尤其敏感。

## 2. 系统总体架构

### 2.1 进程拓扑

```
┌─────────────────────────────────────────────────────────────┐
│                         用户态进程                            │
│                                                              │
│  ┌────────────────────┐        ┌──────────────────────────┐ │
│  │   Xico.app (主应用)  │        │  菜单栏组件 MenuBarExtra  │ │
│  │  SwiftUI + AppKit   │◄──────►│   (同进程或轻量伴随)      │ │
│  └─────────┬──────────┘        └──────────────────────────┘ │
│            │ NSXPCConnection (经过代码签名校验)               │
└────────────┼─────────────────────────────────────────────────┘
             │ XPC (Mach service)
┌────────────▼─────────────────────────────────────────────────┐
│                    特权助手守护进程 (root)                     │
│        com.xico.helper  —  SMAppService.daemon               │
│   · 删除系统级缓存/日志   · 运行维护脚本   · 修复权限           │
│   · 释放可清除空间        · 读硬件传感器     · 受保护路径校验   │
└──────────────────────────────────────────────────────────────┘
             │
       ┌─────▼──────┐   ┌──────────────┐
       │  APFS/文件  │   │  在线定义更新 │ (HTTPS, 规则库/特征库)
       │  系统/TCC   │   │   Backend    │
       └────────────┘   └──────────────┘
```

### 2.2 应用内分层架构（Clean Architecture / 分层 MVVM）

```
┌──────────────────────────────────────────────────────────┐
│  Presentation 层  (SwiftUI Views + ViewModels @Observable) │
│  侧边栏 / 智能扫描 / 各模块面板 / 空间透镜 / 设置 / 菜单栏    │
└───────────────────────────┬──────────────────────────────┘
                            │ 调用 use case，订阅状态流
┌───────────────────────────▼──────────────────────────────┐
│  Domain 层  (业务核心，纯 Swift，无 UI/系统依赖，易测)       │
│  · ScanCoordinator 扫描编排                                 │
│  · CleaningEngine  清理引擎（事务/回滚）                    │
│  · RulesEngine     规则引擎（定义库解释器）                │
│  · SafetyEngine    安全引擎（保护清单/白名单/校验）         │
│  · 各 Module 的 UseCase                                    │
└───────────────────────────┬──────────────────────────────┘
                            │ 通过 protocol 抽象依赖
┌───────────────────────────▼──────────────────────────────┐
│  Modules 层  (插件式特性模块，统一协议)                     │
│  SystemJunk · LargeFiles · SpaceLens · Duplicates ·        │
│  Uninstaller · Privacy · Optimization · Maintenance        │
│  每个模块实现 ScannerModule / CleanerModule 协议           │
└───────────────────────────┬──────────────────────────────┘
                            │
┌───────────────────────────▼──────────────────────────────┐
│  Infrastructure 层  (与系统/外部交互的具体实现)            │
│  FileSystemService · XPCClient · HelperProxy ·            │
│  PersistenceStore · DefinitionsUpdater · Telemetry ·      │
│  PermissionsManager (TCC/FDA) · MetricsSampler           │
└──────────────────────────────────────────────────────────┘
```

**依赖方向**：Presentation → Domain → (protocol) ← Infrastructure。Domain 层不依赖任何具体实现，便于单元测试（可注入 mock 文件系统）。

### 2.3 模块插件化设计

每个功能模块实现统一协议，新增功能不动核心：

```swift
protocol ScannerModule {
    var id: ModuleID { get }
    var metadata: ModuleMetadata { get }            // 名称/图标/分类
    func scan(progress: ProgressReporter) async throws -> ScanResult
}

protocol CleanerModule {
    func plan(from selection: Selection) throws -> CleaningPlan   // 生成可预览计划
    func execute(_ plan: CleaningPlan, via helper: HelperProxy) async throws -> CleaningReport
}
```

`ScanResult` 是统一的树形/分组数据结构，UI 用同一套组件渲染所有模块结果，保证体验一致 + 降低开发量。

## 3. 权限与安全架构（重点）

### 3.1 TCC / 完全磁盘访问（Full Disk Access）

- 要扫描 `~/Library`、Mail、Messages、其他应用容器、`/Library` 等，必须获得 **Full Disk Access（FDA）**。
- App **无法用代码自动授予**，只能引导用户在「系统设置 › 隐私与安全性 › 完全磁盘访问权限」中手动开启。
- 可用 URL 直达该面板：`x-apple.systempreferences:com.apple.preference.security?Privacy_AllFiles`
- 产品需做优雅的「权限缺失」引导页 + 检测已授予状态。

### 3.2 为什么必须非沙盒（关键架构决策）

- App Sandbox 会把应用限制在自己的容器内，无法自由读写其他应用的缓存/系统目录——这与「清理器」的根本职责冲突。
- 因此 Xico **以 Developer ID 分发 + 公证（不进 Mac App Store 主版本）**，关闭 App Sandbox，开启 **Hardened Runtime**。
- 这是 CleanMyMac、DaisyDisk 等的通行做法。（如要进 App Store，只能做功能阉割版，单独工程。）

### 3.3 特权助手（Privileged Helper）

需要 root 的操作（清 `/Library/Caches`、`/private/var/log`、跑维护脚本、修复权限、释放可清除空间、读部分传感器）通过独立守护进程完成：

- **注册**：macOS 13+ 用 `SMAppService.daemon(plistName:)` 注册 LaunchDaemon（替代已弃用的 `SMJobBless`）。
- **通信**：`NSXPCConnection` + 暴露的 Mach service。
- **双向校验（防伪装客户端攻击）**：
  - 助手端校验来连的主应用：用 `kSecCodeInfoIdentifier` / `audit_token` + 代码签名要求（同一 Team ID、签名有效）。
  - 主应用端校验助手身份。
- **最小权限接口**：助手只暴露**白名单化、参数化**的高危操作（如「删除路径列表，且路径必须通过 SafetyEngine 校验」），绝不暴露「执行任意命令」。
- **签名**：助手与主应用同 Team ID 签名，置于 `Contents/Library/LaunchDaemons` 与 `Contents/MacOS`。

### 3.4 安全引擎（SafetyEngine）

清理器的命门。规则：
- **保护清单（永不触碰）**：`/System`、SIP 保护区、`/usr`（除 `/usr/local`）、钥匙串、运行中应用的活跃数据、用户文档/桌面/图片等非缓存目录。
- **路径校验**：任何删除前都要经过 SafetyEngine 规范化路径（解析符号链接、防 `../` 逃逸、防止删到挂载根）。
- **运行态检查**：清某 App 缓存前检测它是否在运行，运行则警告/跳过。
- **可恢复优先**：默认移入废纸篓或可恢复隔离区，提供「撤销」。
- **事务日志**：每次清理写操作日志，崩溃可审计。

详见 [04 · 技术实现要点](04-技术实现要点.md)。

## 4. 并发与性能架构

- 全面使用 **Swift Concurrency**：`async/await`、`actor`（聚合状态隔离）、`TaskGroup`（并行扫描）、`AsyncStream`（进度/结果流）。
- 扫描编排：将顶层目录拆分到 `TaskGroup` 并行枚举，结果由 `actor` 聚合，避免数据竞争。
- **有界并行**：SSD 对并行 IO 友好，但线程过多反而抖动；按 CPU 核数 + IO 特征限流。
- 进度通过 `AsyncStream` 回传，UI 用 `@Observable` 订阅，主线程只做渲染。
- 大结果集用紧凑结构体 + 分页/虚拟化列表渲染，避免一次性持有百万节点（小文件聚合进目录汇总，仅保留目录与大文件明细）。

## 5. 工程结构（建议）

```
Xico/
├── Xico.xcodeproj  (或 Package.swift / Tuist / XcodeGen 生成)
├── App/                      # 应用入口、生命周期、菜单栏
│   ├── XicoApp.swift
│   ├── MenuBar/
│   └── Onboarding/
├── Features/                 # 各功能模块 (Presentation + 模块逻辑)
│   ├── SmartScan/
│   ├── SystemJunk/
│   ├── SpaceLens/
│   ├── LargeFiles/
│   ├── Duplicates/
│   ├── Uninstaller/
│   ├── Privacy/
│   ├── Optimization/
│   └── Maintenance/
├── Domain/                   # ScanCoordinator / CleaningEngine / RulesEngine / SafetyEngine
├── Infrastructure/           # FileSystem / XPC / Persistence / Permissions / Telemetry
├── DesignSystem/             # 设计系统：颜色/字体/组件/动效 (见文档03)
├── Helper/                   # 特权助手守护进程 (独立 target)
│   ├── HelperMain.swift
│   ├── HelperProtocol.swift  # XPC 接口（主应用与助手共享）
│   └── ...
├── Shared/                   # 主应用与助手共享的协议/模型
├── Resources/                # 清理规则库 JSON、本地化、素材
└── Tests/                    # 单元测试 / 集成测试 / 安全回归测试
```

> 可用 **Swift Package Manager** 把 Domain / DesignSystem / 各 Feature 拆成本地 package，强制分层、加速编译、便于测试。
