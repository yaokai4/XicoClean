## 1. 本轮目标

本轮不是简单重画 CPU / 内存卡片，而是重做应用级资源监控的数据链路：

- CPU 面板以 CPU 信息为主，内存面板以内存信息为主。
- 两个面板的每一款应用都同时显示 CPU 与物理内存。
- CPU 默认显示归一化后的 `0–100%`，同时支持原始 `0–N×100%`。
- 应用内存使用 `ri_phys_footprint`，不以虚拟内存或 RSS 冒充。
- Electron / Chrome / XPC 等子进程按最外层 `.app` 归组。
- 首次采样立即显示真实内存，CPU 明确显示“采样中”，不伪造为 0。
- 无详情消费者时停止应用进程采样；打开 CPU 或内存详情时按 1Hz 采样。
- 本地权限不足时可由只读特权助手补充；助手失败时必须快速回退，不能阻塞主线程。

完整设计和执行计划见：

- `docs/superpowers/plans/2026-07-15-xico-precision-monitoring.md`

## 6. 关键口径说明和剩余风险

### 6.1 应用 footprint 与系统“应用内存”不是同一个总量

应用行累加每个成员的 `ri_phys_footprint`；系统卡片的“应用内存”来自 VM 全局分类。两者可能明显不同，尤其是 Electron / Chrome 等共享页较多的应用：逐进程 footprint 求和可能重复体现共享、压缩或账务归属，而系统 VM 分类不会按相同方式重复计算。

因此：

- 应用行可以和 Activity Monitor / `top` 的对应进程 footprint 对比；
- 不能要求所有应用行的 footprint 之和等于系统“应用内存”；
- UI 或文档应明确二者口径不同；
- Task 9 原计划要求针对 Xico、Chrome/Codex 和一个系统 daemon 做现场对比，这一步尚未正式完成。

### 6.2 60 秒温度 / 频率缓存是候选方案，不是最终产品决策

60 秒 TTL 已通过边界测试并降低 CPU 开销，但会让慢变硬件指标最多滞后约一分钟。应用 CPU、应用内存和 per-core 并未降频，仍是 1Hz。交接开发者应在性能门槛和硬件读数新鲜度之间做最终确认，不要把 60 秒 TTL 当成已验收结论。

### 6.3 当前工作区包含大量无关改动

当前 `git status` 约有 111 个变更 / 新文件，包含 SSH、浏览器扩展、扫描、签名脚本等其他用户工作。必须遵守：

- 不使用 `git add -A`；
- 不使用 `git reset --hard` 或 `git checkout --`；
- 不覆盖用户未提交内容；
- 对混合文件使用交互式或 index-only 的逐 hunk 暂存；
- `Package.swift`、`AppModel.swift`、`MenuPanels.swift`、`Components.swift`、`quality_gate.sh` 等尤其不能整文件盲目暂存。

## 7. 建议的最短收尾路径

如果继续本项目，建议按以下顺序，不要再扩大功能范围：

1. 只做一次同实例 scope compare：`steady 60s → memory 60s → cpu 60s`。
2. 同时记录 `LiveMetrics` 和 `ProcessSampler` 各自 CPU，确定剩余成本的真实归属。
3. 若应用流水线仍是主因，优先优化 capture / ownership / aggregation / ranking；不要删除 CPU 面板必要信息，也不要降低应用 CPU / 内存 1Hz。
4. CPU no-panel 增量达到 `<1.5pp` 后，补齐 `ProcessAccuracyBenchmarkTests` 的性能记录。
5. 删除临时 isolation 和 subsystem probe，只保留可长期维护的标准性能入口。
6. 依次运行 focused tests、普通全量测试、真机准确度测试和 quality gate。
7. 重新生成六张截图并人工检查。
8. 逐 hunk 审查 Task 9 改动，更新规格状态，最后再提交。

如果产品方接受当前 `+2.324pp` 而放宽门槛，应先在设计规格和 Task 9 brief 中书面修改验收值，再签字；不能在代码未达原门槛时直接写成通过。
