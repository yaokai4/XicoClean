## 2. 状态与冲突规则

| 状态 | 含义 | 能否计入完成 |
|---|---|---:|
| `verified` | 当前 HEAD 有代码、测试及所需人工/产物证据 | 是 |
| `partial` | 有主体代码，但仍缺要求、失败路径或验收证据 | 否 |
| `open` | 当前明确未实现或有回归 | 否 |
| `external` | 仓库准备可做，但最终闭环需要 Apple、生产密钥、支付/许可服务、服务器或真机 | 否 |
| `superseded` | 被更晚且明确的产品/安全决策覆盖 | 不作为活动任务；保留决策记录 |
| `invalid` | 与安全、平台事实或诚实产品承诺冲突 | 不实现原要求；实现修正后的契约 |
| `historical` | 仅是旧完成声明或旧证据 | 否 |

冲突解决：

- 不恢复已经被用户/实机决策否决的僵尸开关、花哨预设或导致白曝的侧栏材质。
- “100% 可撤销”修正为“默认本地文件清理可撤销；粉碎、清空废纸篓、快照和远端删除明确不可恢复”。
- `.fileSize` 只禁止用于“可释放体积”最终口径；哈希分组可使用逻辑长度。
- Photos Library 只能通过 Photos.framework/用户授权读取，不能裸枚举 `.photoslibrary` 包。
- 天气/时钟维持非目标；除非用户重新改变产品边界。
- 历史 `[ ]` brief 若已有当前提交和报告证据，不重复实现；历史 `[x]` 或“全部完成”若被当前代码推翻，重新打开。
