## 1. 目的与成功定义

Phase 0 的目的不是增加更多视觉效果，而是把 Xico 变成一款可以被独立验证、不会用漂亮完成页掩盖失败的高信任 macOS 工具。完成 Phase 0 后，以下事实必须同时成立：

1. 所有批量操作都能准确表达完整成功、部分完成、失败和取消。
2. 所有不可恢复或高风险操作都经过不可变计划、用户确认授权、执行前身份复核和逐项结果四层边界。
3. 粉碎、卸载、SFTP、主机和隧道不存在绕过确认或身份复核的裸执行入口。
4. 外部深链在用户确认前不会产生网络、磁盘或子进程副作用。
5. 下载组件、更新包、SSH 凭据和发布产物都有明确且 fail-closed 的信任根。
6. 产品内文案、隐私政策、EULA 和真实数据流一致。
7. 只有经过统一 reducer 判定的完整成功才能触发成功庆祝、成功通知和“成功次数”统计。
8. Phase 0 的每条要求都有自动化测试或明确的人工/发布证据，不能用“未发现问题”代替证明。

### 1.1 与最终 95 分目标的关系

用户的最终验收目标不是“综合平均 95 分”，而是以下八个独立维度都达到至少 95/100：

1. 安全、隐私与发布；
2. 可靠性与正确性；
3. 架构与可维护性；
4. 性能与能效；
5. 功能完整性与产品差异化；
6. UX 与信息架构；
7. 视觉、动效与精致度；
8. 无障碍与全球化。

不允许用高分项抵消低分项。每一维都必须同时具备：已实现代码、自动化/人工验收证据、未关闭 Critical/Important 问题为零，以及可复现的评分依据。Phase 0 主要抬升前四项并为后四项建立可信事实基础；Phase 0 关闭不等于宣称最终八项已经达到 95。Phase 1–3 必须继续完成全量文档任务台账中的有效任务，最终才重新评分。

## 4. 总体架构

```text
用户意图
  → 领域 prepare（生成不可变计划和身份快照）
  → UI 展示实际影响
  → AuthorizationIssuer 生成短期能力对象
  → execute 前复核 plan digest、时效、身份与信任边界
  → 领域执行器逐项产生 disposition
  → OperationOutcomeReducer 校验完整性并生成唯一终态
  → OutcomeSideEffectPolicy 决定历史、通知、声音、触觉和撤销
  → TaskOutcomeView 显示 success / partial / failure / cancelled
```

架构由三条互相独立但可组合的主线构成：

1. **Operation Facts**：任务事实和终态 reducer。
2. **Destructive Operation Boundary**：计划、确认授权和目标身份复核。
3. **TrustCore**：联网来源、可执行组件、更新、SSH 和发布信任边界。

## 5. Operation Facts：统一任务事实层

### 5.1 核心类型

建议新增 `Sources/Domain/OperationOutcome.swift`，Domain 不依赖本地化或 SwiftUI。

```swift
public struct OperationKind: RawRepresentable, Hashable, Codable, Sendable {
    public let rawValue: String
}

public enum OperationTerminalStatus: String, Codable, Sendable {
    case success
    case partial
    case failure
    case cancelled
}

public enum OperationIssueCategory: String, Codable, Sendable {
    case permission
    case safetyPolicy
    case notFound
    case identityChanged
    case io
    case network
    case authentication
    case validation
    case timeout
    case unavailable
    case internalInvariant
}

public enum OperationRecoveryHint: String, Codable, Sendable {
    case retry
    case grantPermission
    case installHelper
    case reauthenticate
    case chooseAnotherTarget
    case revealInFinder
    case openSettings
    case manualAction
    case none
}

public struct OperationIssue: Codable, Hashable, Sendable {
    public let code: String
    public let category: OperationIssueCategory
    public let subjectID: String?
    public let recovery: OperationRecoveryHint
    public let retryable: Bool
}

public enum OperationDisposition: Sendable {
    case succeeded
    case unchanged
    case skipped(OperationIssue)
    case failed(OperationIssue)
    case cancelled(OperationIssue?)
}

public struct OperationItemOutcome: Sendable {
    public let subjectID: String
    public let disposition: OperationDisposition
    public let affectedBytes: Int64
}

public struct OperationCounts: Codable, Equatable, Sendable {
    public let requested: Int
    public let succeeded: Int
    public let unchanged: Int
    public let skipped: Int
    public let failed: Int
    public let cancelled: Int
}

public struct OperationOutcome: Codable, Identifiable, Sendable {
    public let id: UUID
    public let parentID: UUID?
    public let kind: OperationKind
    public let status: OperationTerminalStatus
    public let counts: OperationCounts
    public let startedAt: Date
    public let finishedAt: Date
    public let issues: [OperationIssue]
}

public struct OperationResult<Payload: Sendable>: Sendable {
    public let outcome: OperationOutcome
    public let payload: Payload
}

public enum OperationLifecycle<Payload: Sendable>: Sendable {
    case idle
    case running(OperationProgress)
    case cancelling(OperationProgress)
    case terminal(OperationResult<Payload>)
}
```

`OperationOutcome` 的业务初始化器不可公开；只能由 reducer 构造。领域报告继续保留 URL、恢复映射、版本、候选等强类型 payload。

### 5.2 Reducer 不变量

OUT-01：每个请求项必须且只能产生一个 disposition。
OUT-02：缺失或重复结果生成 `internalInvariant`，终态至少为 partial/failure，绝不默认为 success。
OUT-03：`requested == 0` 不构造 success；调用方保持 idle/empty。
OUT-04：用户取消优先。如果执行器在进入终态前接受了取消，终态为 cancelled；计数仍保留取消前真实完成项，当前项可携带取消造成的诚实 issue。
OUT-05：无取消时，全部 succeeded/unchanged 才是 success。
OUT-06：存在真实成功且同时有 failed/skipped 时为 partial。
OUT-07：零成功且存在 failed/skipped 时为 failure。
OUT-08：unchanged 表示目标已满足，不等同于产生了副作用，也不能触发庆祝。
OUT-09：聚合必须按子项 disposition 求和，不能用循环次数或回调次数充当成功数。
OUT-10：重试创建新的 operation ID，并以 parentID 指向上一轮事实；不得篡改旧 outcome。

### 5.3 领域报告迁移

`CleaningReport` 调整为逐项结果：

```swift
public struct CleaningItemResult: Sendable {
    public let itemID: UUID
    public let url: URL
    public let disposition: OperationDisposition
    public let reclaimedBytes: Int64
    public let restorable: RestorableItem?
}

public struct CleaningReport: Sendable {
    public let operation: OperationOutcome
    public let items: [CleaningItemResult]
}
```

迁移期可以保留 `removedCount`、`reclaimedBytes`、`failures` 和 `restorable` 计算属性，但它们不能再作为终态真相。

现有 `DownloadState` 已能表达 completed/failed/quarantined/cancelled，不重写下载生命周期，只适配到 outcome 供统一呈现和诊断。

### 5.4 副作用策略

新增集中式 `OutcomeSideEffectPolicy`：

| 终态 | 真实改变 | 历史 | 成功通知 | 成功声触/粒子 | 失败项重试 |
|---|---:|---|---|---|---|
| success | 有 | 写完整成功记录 | 允许 | 仅安全操作允许 | 无需 |
| success | 无，全部 unchanged | 通常不写清理历史 | 不发送清理完成 | 不庆祝 | 无需 |
| partial | 有 | 写真实成功项并标 partial | 禁止 | 禁止 | 允许 |
| failure | 无 | 不写成功历史 | 禁止 | 禁止 | 允许 |
| cancelled | 有 | 写实际副作用与撤销 receipt，标 cancelled | 禁止 | 禁止 | 可选 |
| cancelled | 无 | 不写 | 禁止 | 禁止 | 可选 |

内部刷新事件与用户成功通知拆分。只要磁盘或远端状态真实改变，可以广播内部 invalidation；这不代表用户操作完整成功。

## 6. 两阶段危险操作边界

### 6.1 API

```swift
func prepare(_ request: Request) async throws -> Plan
func authorize(_ plan: Plan, confirmation: Confirmation) throws -> Authorization
func execute(_ plan: Plan, authorization: Authorization) async -> OperationResult<Payload>
```

计划必须包含：

- `planID`
- `createdAt`、`expiresAt`
- 规范化目标
- 目标身份快照
- 风险等级和可恢复性
- 归属/选择证据
- canonical digest

授权能力对象必须绑定 `planID + digest + nonce + expiresAt`，初始化器只对授权 issuer 可见。它主要防止应用内部误走旁路，不宣称能防御进程内代码注入。

`digest` 使用带 schema 版本的确定性编码计算，不能依赖字典顺序、区域设置或 Swift 默认描述文本。授权 nonce 只能成功消费一次；执行器必须在实际副作用开始前原子消费，重复、过期、摘要不符或 kind 不符全部 fail closed。已经进入终态的 operation 不接受迟到取消，也不能被取消请求改写历史事实。

时效：

- 本地粉碎、卸载、主机和隧道：计划完成后 5 分钟。
- SFTP 删除：30 秒；过期必须重新列目录和确认。
- 执行前仍必须复核身份，未过期不等于身份可信。

### 6.2 本地身份

```swift
LocalFileIdentity(
    device: UInt64,
    inode: UInt64,
    mode: UInt32,
    size: Int64,
    mtimeNanoseconds: Int64,
    hardLinkCount: UInt64
)
```

本地删除至少复核 device/inode/type；粉碎还要复核 hardLinkCount。扫描到执行期间目标变化时跳过新对象，不能按相同路径继续删除。

复核与副作用之间不能重新退化成字符串路径 TOCTOU：本地执行优先持有已复核父目录 fd/目标 fd 并使用 fd-relative syscall；无法保持该约束的执行器必须在最靠近副作用的位置再次复核并失败关闭。

### 6.3 远端身份

```swift
RemoteFileIdentity(
    device: UInt64?,
    inode: UInt64?,
    mode: UInt32,
    size: Int64,
    mtimeNanoseconds: Int64?
)
```

远端服务无法提供稳定身份时不得降级为仅按字符串路径删除；提示用户在服务器终端手动处理。

## 7. 粉碎器设计

### 7.1 准备阶段

SHR-01：每个顶层目标及子项都经过 SafetyEngine。
SHR-02：不跟随符号链接；软链接只代表链接本身。
SHR-03：FIFO、socket、设备等非常规文件整项拒绝。
SHR-04：常规文件 `st_nlink > 1` 默认拒绝，避免覆写未选择的硬链接。
SHR-05：目录预检形成有界身份 manifest；超过条目/manifest 预算时要求拆分，不带未知覆盖面执行。
SHR-06：顶层目录若预检含受保护或无法识别子项，该顶层根不进入授权。

### 7.2 执行阶段

SHR-07：保留现有 fd-relative、`openat(O_NOFOLLOW)`、`unlinkat` 和 inode 复核。
SHR-08：使用注入式 POSIX syscall adapter，便于模拟短写、`EINTR`、ENOSPC、EIO 和 fsync 失败。
SHR-09：使用精确 `pwrite` 循环；remaining 只能按真实写入字节减少。
SHR-10：每一 pass 必须完整写入并成功 `fsync`。
SHR-11：全部 pass 完成、最终 inode/dev/type 复核通过后才允许 unlink。
SHR-12：覆写循环在有界 chunk 间检查取消。接受取消后立即停止后续写入且绝不 unlink；若当前文件已经被改写，标记 `cancelledPossiblyModified`，UI 说明“文件仍在，但内容可能已部分覆写”，并提供重新粉碎剩余目标的动作。取消确认的产品目标为 P95 <500ms；底层阻塞 I/O 无法满足时必须显示“正在停止”而不能宣称已取消。
SHR-13：I/O 中途失败绝不 unlink，并标记 `failedPossiblyModified`；UI 必须说明路径仍在但内容可能已部分覆写。
SHR-14：目录不是可回滚事务；先前成功、后续失败必须形成 partial/failure/cancelled 的真实逐项结果。
SHR-15：粉碎即使完整成功也禁止愉悦型声触强化。

## 8. 卸载器设计

### 8.1 归属模型

```swift
enum UninstallMode { case uninstallApp, cleanLeftovers }
enum SelectionPolicy { case required, recommended, manualOnly, blocked }
enum OwnershipEvidence {
    case exactBundleIDPath
    case signedApplicationGroup
    case launchAgentProgramInsideBundle
    case displayNameHeuristic
    case unverified
}
```

UNI-01：bundle ID 使用严格 ASCII reverse-DNS 解析器；拒绝空段、路径字符、过短弱 token、非法字符和超长分量。
UNI-02：`uninstallApp` 模式下 App 本体为 required，不能取消选择。
UNI-03：App 已不存在时才允许 `cleanLeftovers`。
UNI-04：标准 bundle ID 精确路径可以推荐。
UNI-05：删除 Group Containers 和 LaunchAgents 的 `contains(bundleID)` 匹配。
UNI-06：application groups 从 App 签名 entitlements 精确读取；即使存在也标 manualOnly，因为可能共享。
UNI-07：LaunchAgent 只有 Label 精确且 Program/ProgramArguments 归属该 App 时才推荐。
UNI-08：显示名目录属于 heuristic，默认不选，“全选”不得选择 manualOnly/heuristic。
UNI-09：每个候选展示“为何归属”与恢复方式，不只展示路径。

### 8.2 执行与结果

UNI-10：执行前复核 inode/dev/type 和归属证据。
UNI-11：消费逐项 CleaningReport，不丢弃 failures/restorable。
UNI-12：只移除成功候选；失败候选保留并可重试。
UNI-13：App 本体失败而部分数据成功时，明确显示“数据部分移入废纸篓，但 App 未卸载”。
UNI-14：partial 可撤销已成功的 trash 项。
UNI-15：只有完整成功才清空上下文和进入成功呈现。

## 9. SFTP、主机与隧道

### 9.1 SFTP 删除

SFT-01：结构化列表增加 dev/inode/mode/size/mtime_ns。
SFT-02：计划绑定 host ID、规范 endpoint、固定 host-key 摘要、父目录、原始名称和远端身份。
SFT-03：确认页显示 `user@host:port`、完整路径、类型、大小、指纹摘要和不可恢复性。
SFT-04：普通文件提供“先下载副本”次要动作。
SFT-05：远端从 home 的 `.` 开始逐组件无跟随打开，叶子 `stat(follow_symlinks: false)` 与计划比对。
SFT-06：继续禁止递归目录删除，只允许空目录 `rmdir`。
SFT-07：host key、父目录类型或目标身份变化全部 fail closed。
SFT-08：每个目标只有一个 in-flight 删除，双击不能重复执行。
SFT-09：删除成功但刷新失败应把删除记为成功，并单独报告刷新失败。

远端复核与删除必须在同一个受限 helper 进程内完成，并尽可能使用目录 fd-relative 操作；不能先执行一次 `stat`、退出进程，再按字符串路径启动第二次 `rm`。

### 9.2 SecureHostBinding

Keychain 中的安全连接记录是 endpoint 与凭据的权威来源：

```swift
SecureHostBinding(
    schema: Int,
    revision: UInt64,
    hostID: UUID,
    hostname: String,
    port: Int,
    username: String,
    authKind: SSHAuthKind,
    credentialRef: String?,
    jumpHostBindingID: UUID?,
    pinnedHostKeys: [String],
    networkConfigurationDigest: Data
)
```

SSH-01：`servers.json` 只保存 host ID、当前 binding revision 引用、显示名、分组、图标和其他非安全 UI 元数据。
SSH-02：连接入口只接受 `BoundSSHConnection`，不能再组合任意 ServerHost 与 SSHCredential。
SSH-03：endpoint、username、port、auth ref、jump host 或 pin 变化需要重新确认。
SSH-04：JSON 的 binding revision 缺失、回滚或不存在，以及 legacy JSON 中的 endpoint 等安全字段与 Keychain binding 不一致时，不读取凭据、不联网；迁移/编辑 UI 展示 Keychain 可信值与用户拟议新值的 diff。运行时永远不把 JSON 安全字段当作连接参数。
SSH-05：旧 host 有保存凭据时，升级后默认暂停自动监控/连接，用户核对 endpoint 和指纹后建立 binding。
SSH-06：复用私钥需要用户明确授权，每个 host binding 独立记录该授权。

### 9.3 事务持久化

JSON 与 Keychain 无法形成真正跨存储事务，因此使用版本化绑定和可恢复提交协议：

1. 新建 `binding.<hostID>.<revision>` Keychain 项，不覆盖旧 revision。
2. 原子写入只引用新 revision 的 UI metadata/index。
3. index 持久化成功后，新 revision 才成为 current。
4. 最后垃圾回收旧 binding；失败只产生可重试残留，不破坏 current。
5. 删除主机先完成连接/隧道停止和 jump 依赖检查，再原子删除 index 引用，最后清理 binding/credential。
6. Keychain 清理失败形成 partial 和“重试凭据清理”，不能复活已删除的连接配置。

主机文件存储改为 actor/串行事务 API；写入临时文件、设置目录 `0700`/文件 `0600`、fsync、原子替换、再 fsync 目录。编码或写盘错误必须抛出，不能静默吞掉。

### 9.4 主机和隧道删除

SRV-01：主机确认展示 endpoint、活动连接、活动隧道、jump 依赖和将清理的凭据类型。
SRV-02：明确删除本机配置不会删除远端服务器或远端文件。
SRV-03：私钥按真实 privateKeyRef 清理；共享 ref 使用引用计数。
SRV-04：删除主机时清理 monitor snapshot、console、alert runtime 和 jump 依赖。
SRV-05：`TunnelManager` 提供 `stopAndWait`；必须确认底层进程退出。
SRV-06：SIGTERM 超时后 SIGKILL，再次等待；仍无法确认退出则阻止删除配置。
SRV-07：隧道持久化删除失败时保留“已停止但配置仍存在”的安全可恢复状态。
SRV-08：App 终止统一 stopAllAndWait + disconnectAll，不能只保存下载队列。

### 10.3 外部深链与 SSRF

NET-01：`xico://download` 只能创建 PendingDownloadIntent。
NET-02：确认前不得调用 DownloadManager.add、yt-dlp、aria2、URLSession 抓取或文件写入。
NET-03：预览显示类型、目标 host/port、脱敏 URL、目标目录和预计动作。
NET-04：外部来源默认只接受 HTTPS；magnet 也需要单独确认。
NET-05：拒绝 userinfo、异常端口、混淆 IP、loopback、unspecified、RFC1918、CGNAT、link-local、ULA、multicast、IPv4-mapped private 和公私混合解析。
NET-06：每次 DNS 解析和每一跳重定向复检。
NET-07：外部来源有任务数、频率、单文件/批次大小和并发预算。
NET-08：在可约束 yt-dlp/aria2 全部连接的本地安全代理完成前，外部深链永远不能后台自动开始。
NET-09：浏览器扩展长期迁移 fixed-extension-ID Native Messaging；Shortcuts 使用 App Intent 或 60 秒单次 nonce。
NET-10：自定义 URL scheme 无法证明调用者是浏览器扩展；Native Messaging 完成前，任何 query 参数声称的 extension ID 都不能把来源升级为 trusted browserExtension。
NET-11：内置浏览器只允许受信 HTTPS 官网/支付域参与购买与激活回跳；普通网页、明文 HTTP、跨域跳转或新窗口不能直接触发许可导入。
NET-12：购买前签发绑定 checkout session 的短期一次性 state/nonce；激活回跳必须同时通过 origin、state、时效和一次性消费校验。外部 `xico://activate` 只能进入待复核状态，不能静默激活。

### 10.4 下载组件

CMP-01：Release 中 component catalog 或公钥缺失时直接禁用安装。
CMP-02：同源二进制+checksum 回退只允许显式 DEBUG。
CMP-03：Release endpoint 只允许 HTTPS；file URL 只允许测试/DEBUG 构造器。
CMP-04：组件使用独立签名 key namespace，不与许可、定义或更新混用。
CMP-05：下载必须满足签名清单、精确大小、SHA-256、Mach-O、架构和 archive 安全。
CMP-06：任一失败不得 chmod、不得清 quarantine、不得替换旧组件。
CMP-07：安装使用 private staging、fsync 和原子 replace，并保存可离线复验的签名 receipt；receipt 包含 canonical manifest entry、sequence、key ID 和原 manifest signature/proof，不能只是可改写的 hash 文本。
CMP-08：每次执行前以 App 内置信任根重新验证 receipt proof，再复核文件 hash；receipt 或文件任一被篡改都隔离并要求重新安装。
CMP-09：重定向保持 HTTPS 且受签名清单允许 host 集约束。
CMP-10：Release 自检实际解析 32 字节公钥并验证生产格式 fixture。

## 12. 历史与迁移

### 12.1 Cleaning history

新记录增加：

- outcome status
- requested/succeeded/unchanged/skipped/failed/cancelled counts
- operation kind
- operation/parent ID（仅本地）
- 逐项 restorable receipt

历史层另设 `HistoryOutcomeStatus`，包含当前四种终态以及仅用于迁移的 `legacyUnknown`。旧 JSON 缺少 outcome 时解码为 `legacyUnknown`，不能把它塞入 `OperationTerminalStatus`，也不能倒推成全成功。累计释放可以继续使用旧字节统计，但“成功次数”不能包含 unknown/partial/cancelled。

### 12.2 Server host

旧主机元数据只读取，不自动把它提升为可信 binding。有凭据的 host 首次使用前进入 migration review。迁移中断时保留旧凭据，但保持连接禁用；重新进入 review 可恢复。

### 12.3 Components

旧组件没有 receipt 时不得静默信任。若当前文件 hash 能在有效签名 catalog 中找到，可生成 receipt；否则隔离并要求重新安装。

### 12.4 Updates

legacy alias 和 v2 feed、签名 key、测试 fixture 和发布目录完全隔离。bridge 发布成功后冻结 legacy alias；不得删除已发布客户端仍需访问的精确 URL。

## 13. 本地诊断与隐私

只允许本地聚合：

- operation kind/status
- 成功/失败/跳过/取消数量
- 耗时和字节分桶
- issue code 聚合
- App 版本和 macOS 大版本

禁止默认记录/上传：

- 完整路径、文件名、App 名、bundle ID
- URL、主机名、IP、用户名
- `localizedDescription` 原文
- 跨会话用户行为标识

底层敏感值只以 OSLog private 记录，用户主动导出前提供预览与脱敏。

## 14. 测试设计

### 14.1 Domain reducer

- 全成功、成功+unchanged、部分失败、全部失败、全部跳过。
- 取消前零成功、取消前部分成功。
- 缺失结果、重复结果、requested=0。
- 子任务合并、parent retry。

### 14.2 Cleaning/Shredder/Uninstaller

- Safety 拒绝、路径消失、helper 缺失、普通 I/O 失败、N 项后取消。
- seek/pwrite/fsync 失败不 unlink。
- 短写和 EINTR 正确重试。
- ENOSPC/EIO 返回 failedPossiblyModified。
- 覆写中取消在有界 chunk 后停止、不 unlink，并返回 cancelledPossiblyModified。
- 硬链接零写入、零删除。
- 软链接只删除链接。
- inode/dev/type 改变时零删除新对象。
- 恶意/弱 bundle ID、sibling 冲突、entitlement group、LaunchAgent 不一致。
- App 本体不可取消；App 本体失败而残留成功为 partial。

### 14.3 UI/Side effects

- 四种终态颜色、图标、文案、按钮和 VoiceOver。
- Reduce Motion/Transparency/Contrast/Differentiate Without Color。
- partial/failure/cancelled 不触发成功声触/粒子/通知。
- 同一 operation ID 反馈只消费一次。
- partial 可撤销并仅重试失败项。

### 14.4 Deep link/components/SSH

- 恶意网页触发深链后，确认前网络/磁盘/子进程均为 0。
- IPv4/IPv6 私网、CGNAT、ULA、混合 DNS、redirect、rebinding、超长 URL、userinfo、异常端口和预算。
- 缺 catalog、公钥畸形、签名/过期/回滚/hash/size/arch 错误均零执行、零 quarantine 清除。
- 改 servers.json 的 endpoint/username/port/pin/ref/jump 后，connect/SFTP/tunnel/terminal 全拒绝且不读取凭据。
- legacy migration、Keychain 缺失、迁移中断、真实 host key 变化。
- 共享 privateKeyRef 和 Keychain cleanup partial。
- 任意内置浏览器网页、HTTP/cross-origin 跳转、错误/过期/重放 checkout state 均不能激活许可。
- 伪造 extension ID 的自定义 scheme 仍按 externalURLScheme 处理。

### 14.5 Release E2E

- 干净 checkout、exact signed tag、Universal 架构。
- 主 App/helper 独立 codesign strict、Team ID、bundle ID、entitlements。
- App archive 和 DMG 公证、stapler、spctl。
- N-1→N 标准 Sparkle 更新。
- 使用每个已发布 N-1 二进制内嵌的精确 legacy feed URL 完成 bridge 升级。
- archive 改一字节、错误 key/identity/version 全部拒绝。
- 线上 artifact 回源 hash 与发布 manifest 相同。
- appcast 最后发布。

## 18. 被拒绝的替代方案

### 仅逐功能补丁

可以快速止血，但会继续形成不同的成功判定、确认和网络策略。只允许作为 Batch 2 的临时措施，最终必须进入统一边界。

### 全局持久化事件账本

长期适合下载队列和企业审计，但 Phase 0 会扩大隐私存储面，且无法让粉碎/远端删除变成可回滚事务。暂缓。

### 立即启用完整 App Sandbox/XPC 服务化

隔离最强，但会同时冲击 FDA、system SSH、yt-dlp、helper 和当前打包方式，延迟真实 P0 修复。列入后续安全架构演进。

### 只给 servers.json 加 HMAC

容易出现 JSON、MAC 与 Keychain 跨存储提交不一致，也不能从类型层阻止任意 endpoint 与 credential 组合。选择完整 SecureHostBinding。

### 保留自定义签名但继续使用 `sparkle:edSignature`

与标准 Sparkle archive 签名语义冲突，测试只能证明自定义算法自洽，不能证明工具链互通。选择 legacy bridge + Sparkle 2 v2 feed。

3. **授权重放**：补充 versioned canonical digest 和一次性 nonce 原子消费，避免同一确认能力被重复用于破坏性操作。
4. **TOCTOU**：补充本地 fd-relative 约束，并明确 SFTP 身份复核与删除必须在同一远端 helper 内完成。
5. **历史迁移**：修正 `legacyUnknown` 与四态 `OperationTerminalStatus` 的类型冲突，旧记录只在历史层表达未知。
6. **SSH 权威来源**：澄清 JSON 只引用 binding revision；endpoint、username、credential、pin 和 jump binding 均以 Keychain 安全绑定为准。
7. **主机删除顺序**：修正为先停止连接/隧道并验证退出，再删除 index，防止配置先消失而进程仍运行。
8. **评分口径**：加入八维分别不低于 95 的最终门槛，禁止用平均分掩盖低分项。
9. **回滚安全**：逐批回滚均只能禁用或保留安全状态，不允许恢复 fail-open、裸删除或无签名执行。
10. **更新迁移**：修正“旧客户端改用新 legacy 路径”的不可实现假设，改为保留每个已发布二进制内嵌的精确 feed URL 作为 alias。
11. **组件 receipt**：补充 receipt 自身的签名 proof 和每次执行前离线复验，防止同用户进程同时篡改组件与 hash 文本。
12. **激活回跳**：把 InAppBrowser/`xico://activate` 纳入 TrustCore，要求可信 HTTPS origin 和一次性 checkout state。
13. **实现边界**：本轮仅修改审计/规格文档；产品代码仍保持基线，等待本文书面确认。
